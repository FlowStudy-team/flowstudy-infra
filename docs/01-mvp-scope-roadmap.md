# 01. FlowStudy MVP 范围与版本路线

## 1. 文档目标

本文档用于明确 FlowStudy 的阶段性开发边界，防止项目在早期出现功能膨胀。FlowStudy 的完整愿景包含技术文章、在线判题、AI 侧边栏、学习画像、个性化笔记和 Agent 工作流，但这些内容不能在第一版全部实现。项目应按照 MVP 思路逐步推进，每一版都要有明确目标、功能范围和验收标准。

## 2. 总体版本规划

FlowStudy 建议分为三个主要阶段推进：V1 完成基础学习与判题闭环，V2 完成上下文感知 AI 问答，V3 完成用户画像与个性化笔记生成。

| 版本 | 核心目标 | 关键词 |
|---|---|---|
| V1 MVP | 跑通文章阅读 + 代码提交 + 异步判题主链路 | 用户、文章、题目、提交、判题 |
| V2 AI Assistant | 加入上下文感知 AI 侧边栏 | SSE、上下文、代码报错解释 |
| V3 Personalized Learning | 加入学习画像与自动笔记 | 行为分析、用户画像、Agent、RAG |

## 3. V1 MVP 范围

V1 的目标是证明 FlowStudy 的核心产品形态可以跑通。用户能够登录系统，阅读技术文章，进入章节，查看相关题目，编写代码并提交，系统能够异步判题并返回结果。

V1 必须完成的功能包括用户注册、用户登录、JWT 鉴权、文章列表、文章详情、章节列表、章节详情、题目列表、题目详情、代码模板获取、代码提交、提交记录入库、RabbitMQ 判题任务投递、Judge Service 消费任务、代码运行、结果比对、判题结果回传、Core Service 更新提交状态、前端展示判题结果。

V1 可以暂时不做复杂后台管理系统，文章和题目数据可以先通过初始化 SQL 写入。V1 也可以暂时不做完整 AI 侧边栏，只保留接口预留和页面入口。用户画像、自动笔记、RAG 检索和复杂 Agent 工作流全部放到后续版本。

### 3.1 V1 功能清单

| 模块 | 功能 | 是否必须 |
|---|---|---|
| 用户模块 | 注册、登录、获取当前用户信息 | 必须 |
| 内容模块 | 文章列表、文章详情、章节详情 | 必须 |
| 题目模块 | 题目列表、题目详情、代码模板 | 必须 |
| 提交模块 | 提交代码、查询提交结果 | 必须 |
| 判题模块 | 消费判题任务、运行代码、返回状态 | 必须 |
| 消息队列 | 判题任务投递、判题结果回传 | 必须 |
| Redis | 代码提交接口基础限流 | 建议完成 |
| 前端 | 登录、文章阅读、代码编辑、结果展示 | 必须 |
| AI | 只保留入口，不做完整能力 | 可选 |

### 3.2 V1 不做内容

V1 不实现完整 AI 学习画像，不实现自动生成个性化笔记，不实现复杂 RAG 知识库，不实现多租户，不实现复杂管理员后台，不实现大规模题库导入，不实现排行榜，不实现支付和会员，不实现移动端适配。

这些功能不是不重要，而是不应该进入第一版。第一版的唯一目标是让主链路稳定跑通。

### 3.3 V1 验收标准

V1 完成后，需要满足以下验收条件：用户可以注册并登录；登录后可以查看文章列表；用户可以进入文章详情并阅读章节；章节中可以关联题目；用户可以打开题目并看到题目描述和代码模板；用户可以提交代码；提交后数据库中生成提交记录；Core Service 能够向 RabbitMQ 投递判题任务；Judge Service 能够消费任务并执行代码；判题完成后能够回传结果；前端能够展示 Pending、Running、Accepted、Wrong Answer、Compile Error 等状态。

只要这条链路跑通，即使 UI 还不够精美，AI 功能还不完整，也可以认为 V1 成功。

## 4. V2 AI Assistant 范围

V2 的目标是让 FlowStudy 从普通文章 + OJ 平台升级为“上下文感知学习平台”。这一阶段重点实现 AI 侧边栏问答，让用户可以基于当前文章、章节、题目和代码错误直接提问。

V2 的关键不是接入一个普通聊天接口，而是让 AI 能够自动获得上下文。用户不需要手动复制文章内容、题目描述和报错信息，系统应该自动将这些内容整理进 Prompt 中。

### 4.1 V2 功能清单

| 模块 | 功能 | 说明 |
|---|---|---|
| AI Chat | 侧边栏问答 | 支持用户在学习页面直接提问 |
| SSE | 流式输出 | 提升 AI 回答体验 |
| Context Builder | 上下文拼接 | 拼接文章、章节、题目、代码、错误信息 |
| AI Message | 对话记录保存 | 保存用户问题和 AI 回答 |
| Tracking | AI 提问事件埋点 | 用于后续画像分析 |
| Core Internal API | 提供上下文查询接口 | AI Service 获取业务上下文 |

### 4.2 V2 验收标准

用户在章节或题目页面中可以打开 AI 侧边栏并提问；AI 回答采用 SSE 流式输出；AI 能够知道用户当前阅读章节；AI 能够读取用户最近一次提交代码和判题错误；AI 的回答能够针对具体代码和错误进行解释；用户提问和 AI 回答能够保存到数据库；行为事件能够进入 RabbitMQ 或行为事件表。

## 5. V3 Personalized Learning 范围

V3 的目标是实现 FlowStudy 的核心创新：无感追踪、学习画像和个性化笔记生成。系统不只是回答用户眼前的问题，还要逐步理解用户的学习习惯、错误模式和能力短板。

这一阶段 AI Service 需要异步消费用户行为事件和判题错误事件，对数据进行清洗和结构化分析，形成用户画像。章节结束或周期触发时，系统自动生成 Markdown 学习笔记。

### 5.1 V3 功能清单

| 模块 | 功能 | 说明 |
|---|---|---|
| Behavior Analysis | 用户行为数据清洗 | 分析停留时间、提交频率、错误类型 |
| User Profile | 用户学习画像 | 记录能力标签、薄弱点、代码风格 |
| Note Generation | 个性化学习笔记 | 结合章节内容和用户错误自动生成 |
| RAG | 文章知识检索 | 为 AI 回答和笔记生成提供知识支持 |
| Agent Workflow | 多步骤 AI 工作流 | 分析、总结、生成、校验 |
| Notes Page | 笔记查看页面 | 用户可查看和管理生成的笔记 |

### 5.2 V3 验收标准

系统能够基于用户历史提交和错误生成易错点标签；系统能够识别用户在某章节停留时间较长或频繁提交失败；系统能够生成一份包含章节重点、用户错误总结、改进建议和示例代码的 Markdown 笔记；用户可以在个人中心查看生成的学习笔记；用户画像数据可以持续更新，而不是一次性生成。

## 6. 版本优先级原则

开发过程中必须遵守一个原则：先主链路，再体验优化；先确定性功能，再智能化功能；先单体可跑通，再微服务增强；先本地开发稳定，再考虑生产部署。

具体来说，代码提交与判题链路优先级高于 AI 笔记生成；数据库表结构和 MQ 契约优先级高于页面美化；API 契约优先级高于具体 Controller 实现；Docker Compose 本地环境优先级高于云服务器部署。

## 7. GitHub Project 任务分组建议

GitHub Project 可以按照版本和模块建立任务看板。推荐列为 Backlog、Todo、In Progress、Review、Done。任务标签可以包括 frontend、core、judge、ai、infra、docs、bug、enhancement。

V1 的任务应集中在 infra、core、judge 和 frontend 四类。V2 再集中增加 ai 和 tracking。V3 再增加 profile、rag 和 note-generation。

## 8. 里程碑建议

第一个里程碑是“本地环境跑通”，包括 MySQL、Redis、RabbitMQ 和所有服务的健康检查。第二个里程碑是“内容浏览跑通”，包括用户登录、文章和题目查询。第三个里程碑是“判题链路跑通”，包括提交代码、MQ 投递、Judge 消费和结果回传。第四个里程碑是“前端闭环跑通”，包括从页面提交代码并看到结果。第五个里程碑是“AI 侧边栏跑通”。第六个里程碑是“画像与笔记跑通”。

## 9. 当前最推荐执行顺序

当前阶段建议先完成 flowstudy-infra，包括 docker-compose、数据库初始化 SQL、RabbitMQ 交换机和队列命名。然后开发 flowstudy-core 的用户、文章、题目和提交接口。接着开发 flowstudy-judge 的最小判题 Worker。最后接入 flowstudy-frontend，实现从页面到判题结果展示的完整闭环。AI Service 可以先建项目骨架和健康检查接口，不要过早投入复杂 Agent 逻辑。
