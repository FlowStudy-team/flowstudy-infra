## 1. 项目描述 (Project Description)

**FlowStudy** 是一款面向开发者的“AI 原生交互式学习平台”。 它打破了传统技术博客“作者单向输出、读者被动阅读”的死板模式，将深度技术文章（Java、C++、Go、数据库、算法等）与在线判题系统（OJ）无缝融合。其最大的独创性在于：系统能够在后台无感捕捉读者的学习时长、代码编写风格及报错轨迹，配合上下文感知的 AI 侧边栏助手，最终为每位用户自动提炼出极具个人色彩的专属学习笔记与能力画像。

## 2. 项目愿景 (Project Vision)

- **对于学习者（用户产品价值）**：打造一个绝对沉浸的“心流终端”。让学习者不需要在博客、IDE 和 ChatGPT 之间来回切屏。在这个平台里，代码可以即刻运行，疑问可以随时解答，而你只需专注思考，记录和总结的工作全部交给系统。
- **对于开发者（工程技术价值）**：构建一个具有极高技术壁垒的现代化系统。它不仅要解决传统后端的高并发处理、判题沙箱隔离与中间件异步解耦问题，还要真正落地大型语言模型（LLM）的工程化应用（Agent 编排与 RAG 检索），成为一个教科书级别的全栈/后端架构范例。

## 3. 核心功能模块 (Core Features)

系统由三个核心模块组成，形成一个完整的学习闭环：

### 1. 核心业务模块 (Core Service) —— 系统的“前厅大管家”

这是整个系统的基石，负责所有直接面向用户的常规业务和调度，需要极高的稳定性和并发处理能力。

- **推荐技术栈**：**Java (Spring Boot 3 + Spring MVC + MyBatis-Plus)**
- **核心功能清单**：
  1. **用户与权限体系**：处理用户的注册、登录，以及基于 JWT 或 Session 的安全鉴权。
  2. **内容分发引擎**：负责博客文章、章节结构、题库列表的增删改查与展示渲染。
  3. **高并发网关控制**：接收前端用户提交的代码请求。利用 **Redis + Lua 脚本**实现接口防刷限流，防止恶意提交压垮系统。
  4. **任务投递枢纽**：将用户的代码提交请求、页面停留时长等埋点数据，打包封装后安全地投递到 **RabbitMQ 消息队列**中，实现异步削峰。

### 2. 判题沙箱模块 (Judge Service) —— 系统的“后厨执行者”

这是一个纯粹的、不需要和前端用户打交道的后台服务，专门负责干最耗费 CPU、对安全隔离要求最高的“脏活累活”。

- **推荐技术栈**：**Go (调度层)** + **C/C++ (底层沙箱引擎，推荐直接调用开源实现)**
- **核心功能清单**：
  1. **队列消费**：作为一个安静的 Worker，持续监听 RabbitMQ，从队列中拉取等待判题的代码任务。
  2. **安全隔离运行 (Sandbox)**：将用户的代码扔进极其严格的 Linux 沙箱环境中编译和运行。通过底层机制限制其不能联网、不能读取本地敏感文件，并且精确限制代码的执行时间（防止死循环）和内存使用（防止内存溢出）。
  3. **结果比对与回传**：将代码的实际输出与标准测试用例进行比对，生成最终的运行状态（如 `Accepted` 通过, `Time Limit Exceeded` 超时, `Runtime Error` 报错），并将耗时、内存占用等结果回传给消息队列或数据库。

### 3. AI 分析与网关模块 (AI Agent Service) —— 系统的“最强大脑”

这是系统的灵魂所在，承载了“上下文感知”和“无感追踪生成”的最大亮点。

- **推荐技术栈**：**Python (FastAPI + LangGraph/LangChain)**
- **核心功能清单**：
  1. **侧边栏智能问答 (SSE 流式通信)**：提供与大型语言模型（如 Claude 或 GPT）对接的接口，处理前端侧边栏的对话请求。
  2. **上下文感知拦截**：在向大模型发送问题前，自动将用户当前的“阅读章节”和“最新报错代码”拼接到 Prompt（提示词）中，让 AI 能够精准作答。
  3. **用户行为数据清洗**：异步监听消息队列中用户的报错日志和提问历史，利用小模型或大模型的结构化输出能力，提炼出用户的“易错点”和“代码风格标签”。
  4. **个性化笔记生成引擎**：根据用户累积的数字画像和知识图谱，通过 Agent 工作流（Workflow），在章节结束时全自动生成带有极强个人色彩的 Markdown 学习笔记。



## 4. 核心创新点

**无感追踪与数字画像 (Invisible Tracking & Profiling) —— [核心卖点]**

- **异步采集**：无感记录页面停留时长、AI交互内容、代码提交频率、代码风格、易错点（如边界越界、空指针）。
- **画像更新**：后台 Agent 不断清洗这些行为数据，为用户打上动态能力标签。
- **自动生成笔记**：当章节结束或按周期触发时，结合用户的提问历史和高频错误代码，自动生成一份“带有你个人代码风格和思维习惯”的专属总结笔记。



## 5. 开发指南

### 一、对齐

​	为保证开发进度对齐，风格对齐，提升开发效率，团队成员需要注意以下事项：

#### a. 业务边界与产品形态对齐 (Scope & Product)

在动手前，团队所有人脑子里的产品形态必须是一模一样的，绝不能有人想着做“航母”，有人想着做“竹筏”。

- **锁定 MVP (最小可行性产品) 范围**：明确每一版到底要做什么。例如，第一版只做“用户注册登录 + 浏览文章 + 提交代码并返回判题结果”。至于复杂的 AI 智能笔记、用户动态标签体系，统统放到第二版。
  - 解决方案：使用github project来进行MPV管理
- **梳理核心业务流转图**：不需要画极其复杂的 UML，哪怕在白板上画个草图也行。要明确一条数据从前端输入，经过 Java 调度，落入 MySQL，再扔进 RabbitMQ 让 Go 执行，最后怎么回到前端展示，这条主干链路的每一个节点都要清晰。
  - 解决方案：开发时注意补充文档和注释方便其他人阅读

#### b. 核心“契约”对齐 (API First)

这是微服务并行开发的核心护城河。不同语言栈的开发者，完全依靠“契约”来对话。

- **RESTful API 接口文档**： 强烈建议使用 Apifox、YApi 或 ApiPost。前端和 Java 端必须坐下来，把所有的请求路径、请求参数（Body/Query）、以及全局统一的 `Result<T>` 返回体结构在工具里写死。只要接口文档一发布，前端就可以直接利用工具提供的 Mock 数据开始画页面，后端也可以直接按照接口入参编写 Spring Boot Controller。
  - 解决方案：使用Apifox提前约定好前后端restful 风格接口文档
- **异步消息队列 (MQ) 契约**： Java 和 Go、Python 之间是不通过 HTTP 直接对话的，这就需要对齐 RabbitMQ 的数据结构。你们需要手写几份 JSON 模板文件，明确业务线中 `Exchange` (交换机) 的名称、`RoutingKey` 的规则，以及投递的具体 JSON 字段（比如时间限制、内存限制是传 `int` 还是 `string`）。
  - 解决方案：定义消息队列数据结构
- **数据库表结构与数据字典**： 把咱们之前设计的 `sys_user`、`biz_submit_record` 等建表 SQL 落盘。明确哪些字段允许为空、哪些是唯一索引。这样在使用 MyBatis-Plus 生成代码时，大家对底层数据的认知是一致的。
  - 解决方案：设计数据库表结构，分库分表

#### c. 基础设施基线对齐 (Infrastructure)

不能出现“这代码在我的电脑上明明能跑”这种低级错误。

- **统一本地开发环境 (`docker-compose.yml`)**： 这是正式编码前必须要入库的第一个文件。里面编排好 MySQL（带上初始化的账号密码、建库脚本）、Redis、RabbitMQ。团队每个人克隆代码后的第一件事，就是执行 `docker-compose up -d`，确保所有人的中间件端口、密码 100% 一致。
  - 解决方案：提前定义好本地开发环境，其他人通过docker部署
- **统一大模型 API 接入网关**： 由于 Python 端涉及 LLM 编排，提前申请好大模型的 API Key，并在配置中明确使用环境变量读取，**绝对禁止将 Key 硬编码并提交到代码仓库**。
  - 解决方案：注意api key和环境变量不能公开

#### d. 工程纪律对齐 (Engineering Rules)

- **Git 工作流约定 (Git Flow)**： 确认主分支（`main`）的绝对干净，禁止直接 Push。所有人必须基于 `main` 拉取特性分支（如 `feat/user-login`），开发完成后提交 Pull Request，至少经过一人的 Code Review 才能合并。
  - 解决方案：每个人必须在自己分支上提交代码；同时git commit时需要符合Angular 规范
- **代码风格校验**： 前端配好 ESLint / Prettier，Java 端可以考虑引入 Spotless 插件，Go 使用自带的 `gofmt`。避免在代码合并时，因为缩进是 2 个空格还是 4 个空格而爆发冲突。





## 6. 项目仓库

### (1) .github

* **规定项目总体描述**

* **规范强制模板 (Templates)**

  ​	你可以把我们前面聊过的规范写成模板文件放进去。当别人（或你的队员）新建 Issue 或 PR 时，输入框里会自动带上你写好的格式骨架。

  * `.github/ISSUE_TEMPLATE/bug_report.md`：Bug 反馈模板

  - `.github/pull_request_template.md`：PR 提交模板（比如强制要求勾选“是否已进行本地测试”、“关联了哪个 Issue”）。

* **自动化流水线配置 (GitHub Actions Workflows)**
  - `.github/workflows/deploy.yml`：在这里写脚本，每当有代码合并到 `main` 分支，GitHub 的服务器就会自动拉取你的代码，打包成 Docker 镜像并部署到你的服务器上。

* **社区健康文件 (Community Health Files)**

  - `.github/CODEOWNERS`：用来配置“这块代码归谁管”。如果别人改了 Java 代码提 PR，GitHub 会自动根据这个文件艾特你来做 Code Review。

  - `.github/CONTRIBUTING.md`：贡献指南。告诉新加入的队员，这个项目该怎么跑起来，Git 提交流程是怎样的。
