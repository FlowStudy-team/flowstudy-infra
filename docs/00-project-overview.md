# 00. FlowStudy 项目总览

## 1. 项目定位

FlowStudy 是一款面向开发者的 AI 原生交互式学习平台。它不是单纯的技术博客，也不是单纯的在线判题系统，而是将“技术文章阅读、在线代码练习、AI 辅助问答、学习行为追踪、个性化学习笔记生成”组合成一个完整学习闭环的平台。

项目的核心目标是让学习者不再频繁切换博客、IDE、OJ、ChatGPT 和笔记工具，而是在同一个页面中完成阅读、编码、运行、提问和总结，AI会记录追踪学习者的学习情况并自动生成笔记。对于开发者团队而言，FlowStudy 也是一个覆盖现代后端工程、异步消息系统、判题沙箱、AI Agent、RAG、SSE 流式通信和前端工程化的综合项目。

## 2. 项目愿景

从用户价值看，FlowStudy 希望打造一个“心流式学习终端”。用户阅读技术文章时，可以直接在文章相关位置进行代码练习；代码提交后可以立即获得判题结果；遇到问题时，可以在侧边栏直接询问 AI；学习结束后，系统可以根据用户的行为数据和错误轨迹生成个人化笔记。

从工程价值看，FlowStudy 希望成为一个结构清晰、边界明确、适合团队协作开发的全栈项目。项目采用 Java、Go、Python 和前端技术栈协作，通过 RabbitMQ 完成异步解耦，通过 Redis 完成缓存和限流，通过 MySQL 持久化核心业务数据，通过 AI Agent 服务实现上下文感知问答和学习数据分析。未来其他开发者也可以基于 FlowStudy 快速开发出其他题材的在线学习平台。

## 3. 核心用户场景

用户进入 FlowStudy 后，先登录账号，然后选择一篇技术文章进行学习。文章由多个章节组成，每个章节可以关联若干编程题。用户阅读章节时，前端会记录页面停留时间、滚动进度、代码编辑行为和 AI 提问行为。用户进入题目后，可以使用内置代码编辑器编写代码并提交。提交请求进入 Core Service 后，系统不会同步执行代码，而是将判题任务写入数据库并投递到 RabbitMQ。Judge Service 从消息队列中消费任务，在隔离环境中编译、运行和比对测试用例，然后将结果回传。前端最终展示 Accepted、Wrong Answer、Time Limit Exceeded、Runtime Error 等结果。

当用户遇到错误时，可以直接询问 AI 侧边栏。AI Service 会结合当前文章、当前章节、当前题目、最近提交代码和判题错误信息生成回答。随着用户持续学习，系统会积累用户的提问历史、代码提交记录、错误类型和章节停留时间。后续 AI Service 可以基于这些数据生成用户画像和个性化 Markdown 学习笔记。

## 4. 核心功能模块

### 4.1 Core Service

Core Service 是 FlowStudy 的核心业务服务，使用 Java Spring Boot 3 开发。它负责用户注册登录、权限鉴权、文章管理、章节管理、题目管理、代码提交、提交记录查询、Redis 限流、RabbitMQ 消息投递和判题结果消费。

Core Service 是系统的主业务入口。前端绝大多数普通 HTTP 请求都先进入 Core Service。它不直接执行用户代码，也不直接完成大模型推理，而是负责调度、鉴权、数据落库和异步任务分发。

### 4.2 Judge Service

Judge Service 是判题执行服务，推荐使用 Go 作为调度层，底层沙箱基于成熟开源沙箱实现。它不直接面向前端用户，只负责消费判题任务、运行用户代码、比对测试用例和回传判题结果。

Judge Service 的关键点是安全隔离和资源限制。用户代码必须在受控环境中运行，不能访问宿主机敏感文件，不能联网，不能无限占用 CPU、内存和磁盘资源。MVP 阶段可以先使用 Docker 容器隔离，后续再逐步增强沙箱安全性。

### 4.3 AI Agent Service

AI Agent Service 是 FlowStudy 的创新核心，使用 Python FastAPI 开发，并结 LangGraph 实现 Agent 工作流。它负责 AI 侧边栏问答、SSE 流式输出、上下文拼接、用户行为数据清洗、用户画像更新和个性化笔记生成。

AI Service 不是普通聊天接口，它必须理解用户当前学习上下文。一次 AI 问答至少应该能够感知当前章节内容、当前题目描述、最近一次代码提交和判题错误信息。后续版本中，它还需要结合历史学习行为生成更个性化的学习反馈。

### 4.4 Frontend

Frontend 是 FlowStudy 的交互入口，使用 Vue 3 构建单页应用。它负责文章阅读、Markdown 渲染、题目展示、Monaco Editor 代码编辑、代码提交、判题结果轮询或订阅、AI 侧边栏、学习行为埋点和用户个人中心。

前端不是简单页面展示层，它承担了“沉浸式学习终端”的产品体验。文章区、代码区、判题结果区和 AI 侧边栏之间要形成自然联动。

## 5. 核心创新点

FlowStudy 的最大创新点是无感追踪与数字画像。系统在不打断用户学习的情况下，自动记录用户停留时间、代码提交频率、错误类型、AI 提问内容和代码风格特征。后台 AI 服务持续清洗这些数据，为用户形成动态学习画像。

在传统博客中，读者阅读完文章后很难知道自己真正掌握了什么；在传统 OJ 中，系统只知道用户代码是否通过，很难知道用户为什么错；在普通 AI 问答中，AI 缺少学习上下文。FlowStudy 将这三者结合起来，让系统能够基于真实学习过程生成个性化反馈。

## 6. 技术栈总览

| 模块 | 推荐技术栈 | 主要职责 |
|---|---|---|
| Frontend | Vue 3 , Vite, Monaco Editor | 页面交互、文章阅读、代码编辑、AI 侧边栏 |
| Core Service | Java 17, Spring Boot 3, MyBatis-Plus | 用户、文章、题目、提交、限流、MQ 调度 |
| Judge Service | Go, Docker / Sandbox | 消费判题任务、运行代码、回传结果 |
| AI Service | Python, FastAPI, LangChain / LangGraph | AI 问答、画像分析、笔记生成 |
| Database | MySQL 8.x | 核心业务数据持久化 |
| Cache | Redis 7.x | 登录态辅助、限流、缓存 |
| MQ | RabbitMQ 3.x | 判题任务、行为事件、AI 异步任务 |
| Deployment | Docker Compose / Nginx | 本地开发与后续部署 |

## 7. 当前阶段建设原则

FlowStudy 不应该一开始就做成“大而全”的系统。第一阶段应优先完成用户注册登录、文章阅读、题目展示、代码提交和异步判题这条主链路。AI 画像、RAG、自动笔记生成等高级功能应放到后续版本逐步实现。

当前阶段最重要的不是马上堆功能，而是先统一接口契约、消息契约、数据库结构、环境变量、端口规范和 Git 工作流。只要这些基础契约稳定，前端、Core、Judge 和 AI 服务才能并行开发。
